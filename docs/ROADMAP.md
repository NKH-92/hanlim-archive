# 한림 문서고 관리 시스템 개선 로드맵

작성일: 2026-07-08

## 전제

- 이 시스템의 목적은 **문서가 문서고 어디에 있는지 찾는 것**이다. 검색과 위치확인이 핵심이며, 나머지는 이를 보조한다.
- **무료티어 유지**가 운영 조건이다. 아래 모든 항목은 Cloudflare Workers/D1 무료 한도(요청 10만/일, D1 5GB) 안에서 동작하도록 설계한다. 10인 규모에서는 여유가 크다.
- 소수 인원이 관리하므로 **운영 부담이 늘어나는 기능은 보류**하고, 감사 대응력과 데이터 신뢰성을 우선한다.

## 현재 상태 진단

### 갖춰진 것

- 검색: 오타 허용·부분일치 검색, 검색어 제안, 필터(대분류/태그/구역/상태), 위치순 정렬
- 위치확인: 도면 기반 구역 표시, 랙 지도, 서가 미니맵(열×행 하이라이트)
- 관리: 문서 등록/수정/이동/폐기/복구/완전삭제, 랙·대분류·태그 관리, CSV 가져오기/내보내기, 감사 이력(audit log)
- 계정: 가입 요청 → 관리자 승인, 역할 분리(관리자/일반), CSRF·세션 보안
- **문서 세트**(감사 준비 List → 위치 일괄 조회): 구현 완료, **운영 미배포** (migration 0009)

### 확인된 리스크와 공백

| 항목 | 내용 | 심각도 |
|---|---|---|
| 소스 버전관리 없음 | 이 폴더는 git 저장소가 아님. 운영 중인 코드의 변경 이력·롤백 수단이 없음 | **높음** |
| DB 정기 백업 없음 | 운영 D1의 주기적 백업 절차가 없음 (`backups/` 폴더는 수동) | **높음** |
| 반출 상태 미관리 | 문서를 꺼내가면 시스템 위치와 실물이 어긋남. 감사 시 헛걸음 가능 | 중간 |
| 구역 확장 시 코드 수정 필요 | 구역은 `config.js`에 1~3으로 고정, 구역당 랙 15개 상한. 랙 추가는 UI로 가능하나 구역 신설은 불가 | 중간(증설 전까지 낮음) |
| 외부 CDN 의존 | 폰트·아이콘을 jsdelivr/cdnjs에서 로드, CSP 미적용 (기존 배포 문서에도 기재된 사항) | 중간 |
| 도면 영역 편집 UI 없음 | `floor_plan_regions`는 SQL로만 수정 가능 | 낮음 |

## 1단계 — 기반 정리와 세트 배포 (즉시, 반나절)

지금 당장 하는 일. 기능 추가보다 앞서야 한다.

1. **git 저장소 초기화** 후 현재 상태를 첫 커밋으로 고정. GitHub 비공개 리포에 푸시해 두면 백업을 겸한다.
2. **세트 기능 운영 배포**
   ```powershell
   cd .\cloudflare-app
   npm run db:migrate:remote
   npm run deploy
   ```
   배포 후 운영 URL에서 로그인 → 세트 생성 → 일괄 추가 스모크 테스트.
3. **실전 세트 1개 등록**: 실제 감사 준비문서 List를 등록해 보고, 표·인쇄물이 실무에 맞는지 사용자(파트원) 피드백 수집.

완료 기준: 운영 URL에서 세트 기능 동작, 코드가 git으로 관리됨.

## 2단계 — 감사 대응 완성 (단기, 1~2주 내 순차 적용)

세트 기능을 "감사 실무 도구"로 완성한다. 각 항목은 반나절 내외의 작은 작업.

1. **인쇄물 실무 양식화**: 인쇄 시 체크 열(□ 꺼냄 / □ 반납), 출력 일시, 확인자 서명란 추가 → 종이 한 장으로 회수 대장 역할.
2. **세트 CSV 내보내기**: 위치 목록을 엑셀로 제출·공유할 수 있게.
3. **세트 변경 이력**: 누가 언제 어떤 문서를 추가/제외했는지 기록 (기존 audit log 패턴 재사용).
4. **미등록 번호 후속 처리**: 일괄 추가에서 "찾지 못한 번호"를 클릭하면 해당 번호가 채워진 문서 등록 화면으로 이동.

완료 기준: 감사 당일 세트 화면 + 인쇄물만으로 문서 회수와 기록이 끝남.

## 3단계 — 반출·반납 관리 (중기, 2~4주)

이 시스템 신뢰성의 핵심 과제. 시스템이 아는 위치와 실물 위치가 어긋나는 유일한 경로가 "누가 꺼내감"이다.

- 문서 상태에 **반출 중** 추가: 반출자·반출일·용도 기록, 반납 처리.
- 검색 결과·문서 카드·세트 목록에 반출 중 표시. 세트 조회 시 "반출 중 N건 — 회수 필요" 경고.
- 반출 이력은 기존 이동 이력처럼 타임라인으로 조회.

완료 기준: 감사 준비 세트를 열면 "랙에 없는 문서"가 즉시 드러나고 누구에게 있는지 보임.

## 4단계 — 물리 확장 대비 (문서고 증설 확정 시 착수)

랙은 지금도 UI로 추가 가능(구역당 최대 15개). 이 단계는 **구역 자체가 늘어날 때** 필요하다.

1. 구역 동적 관리: 하드코딩된 `RACK_ZONES [1,2,3]` 제거, 구역 CRUD 화면 (DB의 `floor_plan_regions`와 연동).
2. 구역당 랙 상한(15) 설정값으로 전환.
3. 도면 영역 편집 UI: 도면 위 구역 박스 위치·크기를 관리자 화면에서 조정.
4. 도면 이미지 교체 절차 문서화 (정적 자산이라 재배포 필요).

완료 기준: 문서고 증설 시 코드 수정 없이 관리자 화면에서 구역·랙 반영 가능.

## 5단계 — 보안·운영 안정화 (상시, 다른 단계와 병행 가능)

1. **주기 백업 자동화**: GitHub Actions 스케줄로 `wrangler d1 export` 주 1회 실행 → 리포에 보관 (1단계 git화 선행 필요).
2. **폰트·아이콘 self-host + CSP 적용**: 외부 CDN 의존 제거 (기존 배포 문서의 미결 항목).
3. **로그인 시도 제한**: D1 카운터 기반 계정 잠금(예: 10분 내 5회 실패 시 잠금)으로 무차별 대입 방어.
4. 무료티어 사용량 점검 방법 문서화 (Cloudflare 대시보드 지표 확인 주기).

## 운영 원칙 (변경 시 항상)

- 변경 전 `npm run check` + `npm test`, 로컬 `wrangler dev` 확인 후 배포.
- 스키마 변경은 반드시 migration 파일로 (수동 SQL 금지, 초기 관리자 생성 제외).
- 기능 추가보다 **감사 시나리오에서의 소요 시간 단축**을 판단 기준으로 삼는다.
